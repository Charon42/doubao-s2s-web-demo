# 豆包端到端实时语音 S2S Demo 代码讲解

本文档解释当前项目的实现逻辑和关键代码。项目核心目标是：浏览器采集麦克风音频，发送给 FastAPI 后端；后端用火山引擎豆包端到端实时语音的 WebSocket 二进制协议转发音频；豆包返回 ASR 文本、AI 文本和 TTS 音频；前端展示文本并播放音频。

## 1. 项目结构

```text
backend/
  main.py              FastAPI 服务入口，提供页面和 /ws/call WebSocket
  doubao_realtime.py   豆包实时语音 WebSocket 协议封装
  memory.py            简单对话文本内存
  .env                 本地密钥和豆包配置
  .env.example         配置模板

frontend/
  index.html           页面结构
  app.js               录音、WebSocket、播放、事件处理
  style.css            页面样式
```

整体数据流：

```text
麦克风
  -> frontend/app.js 转成 16kHz mono PCM16
  -> 浏览器 WebSocket /ws/call
  -> backend/main.py
  -> backend/doubao_realtime.py 封装豆包二进制包
  -> 豆包 WebSocket
  -> 豆包返回 ASR / Chat / TTS 事件
  -> 后端解析成前端 JSON 事件
  -> 前端显示文本、播放 PCM 音频
```

## 2. 后端入口：main.py

`backend/main.py` 做两件事：

1. 提供静态网页。
2. 提供 `/ws/call` WebSocket 作为浏览器和豆包之间的中转。

关键代码：

```python
app.mount("/static", StaticFiles(directory=FRONTEND_DIR), name="static")

@app.get("/")
async def index() -> FileResponse:
    return FileResponse(FRONTEND_DIR / "index.html")
```

这让浏览器访问 `http://127.0.0.1:8000/` 时拿到 `frontend/index.html`，并能加载 `/static/app.js` 和 `/static/style.css`。

### /ws/call 的作用

```python
@app.websocket("/ws/call")
async def call_ws(websocket: WebSocket) -> None:
    await websocket.accept()
```

浏览器点击开始通话后，会连接这个 WebSocket。后端随后创建豆包客户端：

```python
doubao = DoubaoRealtimeClient()
await doubao.connect()
```

连接成功后，后端给前端发 `ready`：

```python
await websocket.send_json({
    "type": "ready",
    "session_id": doubao.session_id,
    "input_sample_rate": doubao.config.input_sample_rate,
    "output_sample_rate": doubao.config.output_sample_rate,
})
```

前端据此知道后端和豆包已经准备好，同时知道返回音频采样率是 24000Hz。

### 两个并发任务

`main.py` 里有两个异步任务：

```python
browser_to_doubao()
doubao_to_browser()
```

`browser_to_doubao()` 负责接收浏览器发来的音频 bytes：

```python
if "bytes" in message and message["bytes"]:
    await doubao.send_audio(message["bytes"])
```

`doubao_to_browser()` 负责读取豆包事件，再转发给前端：

```python
async for event in doubao.events():
    await websocket.send_json(event)
```

这就是后端“中转”的核心：浏览器不直接连豆包，因此 API Key 不会暴露给前端。

## 3. 豆包协议封装：doubao_realtime.py

`backend/doubao_realtime.py` 是项目里最关键的文件。它负责：

- 读取 `.env`
- 构造豆包握手 Header
- 编码豆包自定义二进制协议
- 解码豆包返回的二进制包
- 发送 StartConnection / StartSession / TaskRequest 等事件
- 把豆包事件转换成前端能理解的 JSON

## 4. 配置读取

配置集中在 `DoubaoRealtimeConfig`：

```python
@dataclass(frozen=True)
class DoubaoRealtimeConfig:
    ws_url: str
    app_id: str
    access_key: str
    resource_id: str
    app_key: str
    model_version: str
    speaker: str
    bot_name: str
    system_role: str
    speaking_style: str
    greeting: str
    input_sample_rate: int
    output_sample_rate: int
```

`.env` 中的关键项包括：

```env
DOUBAO_WS_URL=wss://openspeech.bytedance.com/api/v3/realtime/dialogue
DOUBAO_APP_ID=
DOUBAO_ACCESS_KEY=
DOUBAO_RESOURCE_ID=volc.speech.dialog
DOUBAO_APP_KEY=
DOUBAO_MODEL_VERSION=1.2.1.1
DOUBAO_SPEAKER=zh_female_vv_jupiter_bigtts
DOUBAO_BOT_NAME=
DOUBAO_SYSTEM_ROLE=
DOUBAO_SPEAKING_STYLE=
DOUBAO_GREETING=
```

鉴权 Header 由 `build_headers()` 生成：

```python
def build_headers(config: DoubaoRealtimeConfig, connect_id: str) -> dict[str, str]:
    return {
        "X-Api-App-ID": config.app_id,
        "X-Api-Access-Key": config.access_key,
        "X-Api-Resource-Id": config.resource_id,
        "X-Api-App-Key": config.app_key,
        "X-Api-Connect-Id": connect_id,
    }
```

这些 Header 只存在后端，不会发送给浏览器。

## 5. 豆包二进制协议

豆包端到端实时语音使用自定义二进制协议，不是普通 JSON WebSocket。

当前代码里的包结构是：

```text
4 字节 header
event id
可选 session_id
可选 connect_id
payload_size
payload
```

### encode_packet()

`encode_packet()` 把事件和 payload 编码成豆包需要的 bytes：

```python
def encode_packet(...):
    body = bytearray()
    body.extend(event.to_bytes(4, "big", signed=True))
    ...
    body.extend(len(payload).to_bytes(4, "big"))
    body.extend(payload)
```

第一段 4 字节 header：

```python
return bytes([
    (VERSION << 4) | HEADER_SIZE_UNITS,
    (message_type << 4) | FLAG_WITH_EVENT,
    (serialization << 4) | compression,
    0x00,
]) + bytes(body)
```

几个重要字段：

- `VERSION = 0x1`：协议版本。
- `MSG_FULL_CLIENT = 0x1`：客户端 JSON 事件。
- `MSG_AUDIO_ONLY_CLIENT = 0x2`：客户端音频事件。
- `FLAG_WITH_EVENT = 0x4`：包里包含 event id。
- `SERIALIZATION_JSON = 0x1`：payload 是 JSON。
- `SERIALIZATION_NONE = 0x0`：payload 是原始音频 bytes。

### decode_packet()

`decode_packet()` 反向解析豆包返回包：

```python
message_type = (data[1] >> 4) & 0x0F
flags = data[1] & 0x0F
serialization = (data[2] >> 4) & 0x0F
compression = data[2] & 0x0F
```

然后读取 event、session_id、payload_size、payload，最后返回：

```python
DoubaoPacket(
    message_type=message_type,
    event=event,
    session_id=session_id,
    payload=payload,
)
```

这让后续代码可以按 `packet.event` 判断豆包返回了什么。

## 6. 连接豆包：StartConnection 和 StartSession

`DoubaoRealtimeClient.connect()` 是豆包连接流程：

```python
self.ws = await websockets.connect(self.config.ws_url, additional_headers=headers)
```

然后发送 `StartConnection = 1`：

```python
await self._send_event(EVENT_START_CONNECTION, {})
packet = await self._recv_packet()
```

收到 `ConnectionStarted = 50` 后，再发送 `StartSession = 100`：

```python
start_session_payload = self._start_session_payload()
await self._send_event(EVENT_START_SESSION, start_session_payload)
```

收到 `SessionStarted = 150` 后，说明会话初始化成功。

如果 `.env` 配置了 `DOUBAO_GREETING`，代码会发送 `SayHello = 300`：

```python
if self.config.greeting:
    await self._send_event(EVENT_SAY_HELLO, {"content": self.config.greeting})
```

### StartSession payload

`_start_session_payload()` 构造会话配置：

```python
{
    "asr": {
        "language": "zh-CN",
    },
    "tts": {
        "speaker": self.config.speaker,
        "audio_config": {
            "channel": 1,
            "format": "pcm_s16le",
            "sample_rate": 24000,
            "bits": 16,
        },
    },
    "dialog": {
        "bot_name": self.config.bot_name,
        "system_role": self.config.system_role,
        "speaking_style": self.config.speaking_style,
        "extra": {
            "model": self.config.model_version,
            "input_mod": "audio",
        },
    },
}
```

这里有两个重点：

- 输入音频是前端传来的 16kHz PCM16。
- 输出音频要求豆包返回 `pcm_s16le`、单声道、24000Hz，前端按这个格式播放。

## 7. 前端录音和音频上传

前端在 `frontend/app.js` 中用浏览器 API 采集麦克风：

```javascript
mediaStream = await navigator.mediaDevices.getUserMedia({
  audio: {
    channelCount: 1,
    echoCancellation: true,
    noiseSuppression: true,
    autoGainControl: true,
  },
});
```

然后创建音频处理节点：

```javascript
inputContext = new AudioContext();
sourceNode = inputContext.createMediaStreamSource(mediaStream);
processorNode = inputContext.createScriptProcessor(4096, 1, 1);
```

浏览器采集到的是 Float32 PCM，范围是 `-1.0 ~ 1.0`。豆包要求输入是 `int16` PCM，所以需要转换：

```javascript
function floatToPcm16(float32) {
  const pcm = new Int16Array(float32.length);
  ...
  pcm[i] = sample < 0 ? sample * 0x8000 : sample * 0x7fff;
  return pcm;
}
```

如果浏览器采样率不是 16000Hz，还会降采样：

```javascript
const pcm16 = floatToPcm16(downsampleTo16k(mono, inputContext.sampleRate));
ws.send(pcm16.buffer);
```

## 8. 后端发送音频：TaskRequest = 200

前端发来的音频进入后端：

```python
await doubao.send_audio(message["bytes"])
```

`send_audio()` 会把音频缓存起来，并按 640 字节切包：

```python
PCM_16K_20MS_BYTES = 640
```

为什么是 640？

```text
16000 samples/s * 0.02s = 320 samples
320 samples * 2 bytes/int16 = 640 bytes
```

每个 20ms 音频包通过 `TaskRequest = 200` 发给豆包：

```python
packet = encode_packet(
    event=EVENT_TASK_REQUEST,
    message_type=MSG_AUDIO_ONLY_CLIENT,
    serialization=SERIALIZATION_NONE,
    payload=chunk,
)
```

音频 payload 是原始 PCM bytes，不是 JSON。

## 9. 豆包事件解析

后端的 `events()` 持续读取豆包 WebSocket：

```python
async for message in self.ws:
    packet = decode_packet(message)
    event = self._packet_to_frontend_event(packet)
```

主要事件：

```text
450 ASRInfo       用户开始说话
451 ASRResponse   用户识别文本
459 ASREnded      用户说话结束
352 TTSResponse   AI 音频
550 ChatResponse  AI 文本
559 ChatEnded     AI 文本结束
359 TTSEnded      TTS 音频结束
599 Error         错误
```

### 用户文本

文档说明 `ASRResponse=451` 的文本在：

```text
payload.results[0].text
```

代码用 `_extract_asr_text()` 专门读取这个字段：

```python
def _extract_asr_text(payload):
    results = payload.get("results")
    if isinstance(results, list) and results:
        first = results[0]
        if isinstance(first, dict):
            return first["text"]
```

如果文本为空，不发给前端，避免页面出现空回车。

### AI 文本

`ChatResponse=550` 只读取：

```text
payload.content
```

代码：

```python
content = str(payload.get("content") or "").strip()
if not content:
    return None
return {"type": "assistant_text", "text": content}
```

这避免把状态事件误当成文本。

### AI 音频

`TTSResponse=352` 的 payload 是音频二进制：

```python
audio = packet.payload
return {
    "type": "audio",
    "audio": base64.b64encode(audio).decode("ascii"),
    "sample_rate": 24000,
}
```

后端通过 JSON 发给前端，所以二进制音频需要 base64。

## 10. 前端播放 PCM 音频

前端收到：

```json
{"type":"audio","audio":"...base64...","sample_rate":24000}
```

先 base64 解码：

```javascript
function base64ToArrayBuffer(base64) { ... }
```

再按 little-endian int16 解析：

```javascript
function pcm16ToFloat32(arrayBuffer) {
  const view = new DataView(arrayBuffer);
  const floats = new Float32Array(Math.floor(arrayBuffer.byteLength / 2));
  for (let i = 0; i < floats.length; i += 1) {
    floats[i] = view.getInt16(i * 2, true) / 0x8000;
  }
  return floats;
}
```

这里 `true` 表示 little-endian。然后创建 `AudioBuffer`：

```javascript
const buffer = outputContext.createBuffer(1, floats.length, playbackSampleRate);
buffer.copyToChannel(floats, 0);
```

最后播放：

```javascript
const source = outputContext.createBufferSource();
source.buffer = buffer;
source.connect(outputContext.destination);
source.start();
```

代码还检查了 OGG Opus 头：

```javascript
if (bytes[0] === 0x4f && bytes[1] === 0x67 && bytes[2] === 0x67 && bytes[3] === 0x53)
```

如果返回是 `OggS`，说明服务端没有按 `pcm_s16le` 返回，前端不会把它当 PCM 播放。

## 11. 文本显示逻辑

前端根据后端事件类型更新文本区域。

### 用户文本

```javascript
if (msg.type === "user_text") {
  if (msg.is_interim) {
    interimUserText = text;
  } else {
    finalUserText += `${text}\n`;
    interimUserText = "";
  }
  renderUserText();
}
```

临时识别文本只覆盖显示，不一直换行。最终文本才追加换行。

### AI 文本

```javascript
if (msg.type === "assistant_text") {
  assistantText.textContent += msg.text || "";
}
```

AI 文本是流式 delta，所以不能每个 delta 后都加换行。只有收到：

```javascript
if (msg.type === "assistant_text_done") {
  assistantText.textContent += "\n";
}
```

才表示一轮 AI 文本结束。

## 12. 打断机制

打断分后端和前端两部分。

### 后端打断服务端

当豆包发来 `ASRInfo=450`，说明用户开始说话：

```python
if packet.event == EVENT_ASR_INFO:
    self.reply_id += 1
    await self._send_event(EVENT_CLIENT_INTERRUPT, {"session_id": self.session_id})
    yield {"type": "user_speech_start", "reply_id": self.reply_id}
```

后端会做两件事：

1. 给豆包发送 `ClientInterrupt=515`，让服务端停止当前回复。
2. 给前端发送 `user_speech_start`，让前端立刻停止本地正在播放的旧音频。

### 前端打断播放

前端收到 `user_speech_start`：

```javascript
if (msg.type === "user_speech_start") {
  currentReplyId = Math.max(currentReplyId + 1, msg.reply_id || 0);
  stopPlayback();
}
```

`stopPlayback()` 会：

- 清空音频队列。
- 停止当前所有 `AudioBufferSourceNode`。
- 重置播放状态。

```javascript
function stopPlayback() {
  audioQueue = [];
  audioPlaying = false;
  for (const source of activeSources) {
    source.stop();
    source.disconnect();
  }
  activeSources.clear();
}
```

### reply_id 的作用

`reply_id` 用来区分新旧轮次音频。

前端收到音频时：

```javascript
const replyId = msg.reply_id || currentReplyId;
if (replyId < currentReplyId) {
  return;
}
```

如果旧一轮的音频晚到了，前端会丢弃它，不会插队播放。

## 13. memory.py 的作用

`backend/memory.py` 很简单，只保存已确认的用户文本和 AI 文本：

```python
memory.add("user", event.get("text", ""))
memory.add("assistant", event.get("text", ""))
```

当前版本没有把 memory 再发给豆包做上下文，它主要用于后续扩展，比如保存对话记录、写日志、做多轮上下文。

## 15. 一次完整通话的时序

```text
1. 用户点击“开始通话”
2. 前端请求麦克风权限
3. 前端连接 /ws/call
4. 后端连接豆包 WebSocket
5. 后端发送 StartConnection=1
6. 豆包返回 ConnectionStarted=50
7. 后端发送 StartSession=100
8. 豆包返回 SessionStarted=150
9. 后端通知前端 ready
10. 前端持续发送 PCM16 音频
11. 后端按 640 bytes 切包发送 TaskRequest=200
12. 豆包返回 ASRResponse=451
13. 页面显示用户文本
14. 豆包返回 ChatResponse=550
15. 页面显示 AI 文本
16. 豆包返回 TTSResponse=352
17. 页面播放 AI 音频
18. 用户再次说话时，ASRInfo=450 触发打断
```

## 16. 常见问题定位

如果没有用户文本：

- 看后端是否收到 `ASRResponse=451`。
- 检查 payload 是否真的有 `results[0].text`。
- 如果字段不一致，需要按官方文档调整 `_extract_asr_text()`。

如果 AI 文本错乱：

- 看是否只在 `ChatResponse=550` 读取 `payload.content`。
- 状态事件 350、351、359、459、559 不应该当文本。

如果音频噪声大：

- 确认 StartSession 里 `tts.audio_config.format = "pcm_s16le"`。
- 确认 sample rate 是 24000。
- 确认前端用 little-endian `getInt16(offset, true)`。
- 如果返回 `OggS`，说明服务端返回的是 OGG Opus，不应走 PCM 播放。

如果打断无效：

- 检查后端是否收到 `ASRInfo=450`。
- 检查后端是否发送 `ClientInterrupt=515`。
- 检查前端是否收到 `user_speech_start`。
- 检查旧 `reply_id` 的音频是否被丢弃。

## 17. 设计取舍

这个 Demo 是最小可运行版本，所以有几个有意简化：

- 前端使用 `ScriptProcessorNode`，没有换成更现代的 `AudioWorklet`。
- 没有数据库和登录系统。
- memory 只保存本地文本，没有做持久化。
- 协议字段中仍有少量 TODO，需要按你手里的豆包官方文档最终确认。

核心原则是：前端只管录音、展示和播放；后端负责密钥、豆包连接和协议转换。
